## Descrição

Implementar cobertura de testes para `src/pages/Dashboard.tsx` (25KB, ~1000 linhas), cobrindo cálculos de KPIs financeiros e integração com Realtime:
- KPI calculation (NCG, DIO, DSO, DPO, MBL)
- State management (filters, sorting, pagination)
- Realtime updates (contratos, requisições)
- Chart rendering (Recharts integration)
- Filter/sort logic

### Contexto

Dashboard é a visão executiva principal de Lexflow — qualquer cálculo de KPI errado direciona decisões estratégicas incorretas (alocação de capital, renegociação de contratos, etc).

**Impacto**: Sem testes, regressões em cálculos financeiros podem passar para produção desapercebidas.

**Business Impact**: Decisões baseadas em KPIs incorretos = custos operacionais altos.

## Acceptance Criteria

### AC 1: Unit Tests para KPI Calculation
- [ ] Teste: NCG (Necessidade de Capital de Giro) calculado corretamente
  - Given: Dados de contratos/requisições conhecidos
  - When: Dashboard calcula NCG
  - Then: NCG = (DIO + DSO - DPO) × valor médio diário
- [ ] Teste: DIO (Days Inventory Outstanding) calculado corretamente
  - Given: Data de criação e entrega de requisições
  - When: DIO é calculado
  - Then: DIO = dias médios em inventário
- [ ] Teste: DSO (Days Sales Outstanding) calculado corretamente
  - Given: Data de faturamento e pagamento
  - When: DSO é calculado
  - Then: DSO = dias médios para receber
- [ ] Teste: DPO (Days Payable Outstanding) calculado corretamente
  - Given: Data de compra e pagamento a fornecedor
  - When: DPO é calculado
  - Then: DPO = dias médios para pagar
- [ ] Teste: MBL (Margem Bruta Líquida) calculado corretamente
  - Given: Receita e custos conhecidos
  - When: MBL é calculado
  - Then: MBL = (receita - custos) / receita

**Coverage Target**: 90% de KPI calculation logic

### AC 2: Unit Tests para State Management
- [ ] Teste: Filters aplicados corretamente
  - Given: Dashboard com múltiplos contratos
  - When: Filtro por status="vigente" é aplicado
  - Then: Apenas contratos vigentes são exibidos
- [ ] Teste: Sorting funciona
  - Given: Contratos desordenados
  - When: Sortear por "data_vencimento DESC"
  - Then: Contratos mais próximos de vencer aparecem primeiro
- [ ] Teste: Pagination funciona
  - Given: 50 contratos, page size = 10
  - When: Navegar para página 2
  - Then: Itens 11-20 são exibidos

**Coverage Target**: 85% de state management

### AC 3: Integration Tests para Realtime
- [ ] Teste: Dashboard subscriba em Realtime
  - Given: Dashboard montado
  - When: Novo contrato é criado no BD
  - Then: Dashboard recebe update via WebSocket (< 100ms)
- [ ] Teste: Chart atualiza em tempo real
  - Given: Realtime subscription ativa
  - When: Novo contrato afeta KPI
  - Then: KPI no chart é atualizado automaticamente

**Coverage Target**: 75% de Realtime integration

### AC 4: Unit Tests para Chart Rendering
- [ ] Teste: Recharts componentes renderizam
  - Given: KPI data válido
  - When: Chart é montado
  - Then: Chart é renderizado sem erro
- [ ] Teste: Chart responsivo
  - Given: Container width = 300px
  - When: Chart é montado
  - Then: Chart redimensiona e permanece legível

**Coverage Target**: 80% de chart logic

### AC 5: Error Handling Tests
- [ ] Teste: Missing data é tratado gracefully
  - Given: KPI data incompleto
  - When: Dashboard renderiza
  - Then: "Dados insuficientes" é exibido, não crash
- [ ] Teste: Realtime disconnect é tratado
  - Given: Realtime subscription
  - When: Conexão cai
  - Then: Dashboard mostra "offline" indicator + tenta reconectar

**Coverage Target**: 85% de error paths

### AC 6: Mocks Configurados
- [ ] Mock de TanStack React Query
- [ ] Mock de Supabase Realtime
- [ ] Mock de Recharts
- [ ] Fixtures de KPI data (diversos cenários)

**Coverage Target**: 100% de dependencies mockadas

- [ ] Coverage report: >= 80% para Dashboard.tsx

- [ ] CI/CD pipeline passa
