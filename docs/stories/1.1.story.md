## Descrição

Implementar sistema de notificações automáticas para eventos críticos de contrato, utilizando PostgreSQL triggers para detecção de eventos, Supabase Realtime para WebSocket updates, e Edge Functions para delivery multi-canal (email, WhatsApp, SMS).

### Contexto

Sistema de notificações é crítico para operações financeiras de Lexflow:
- **Alertas de vencimento**: Notificar 30, 15, 7 dias antes do fim de contrato
- **Mudanças de status**: Notificar quando contrato muda de rascunho → aprovado → vigente → encerrado
- **Novas obrigações**: Notificar quando nova obrigação contratual é criada
- **Preferências do usuário**: Respeitar canal preferido (email, WhatsApp, ou desabilitar)

## Acceptance Criteria

### AC 1: Tabela `notifications` criada com RLS e Realtime
- [ ] Tabela `notifications` criada em PostgreSQL
  - Campos: `id`, `organization_id`, `user_id`, `tipo`, `titulo`, `mensagem`, `referencia_id`, `referencia_tipo`, `lida`, `created_at`
  - Constraint: `tipo IN ('vencimento', 'status_change', 'aprovacao', 'obrigacao', 'geral')`
  - Constraint: `referencia_tipo IN ('contrato', 'obrigacao', 'fornecedor', NULL)`
- [ ] RLS policies ativas: `users_see_own_notifications`, `users_update_own_notifications`
- [ ] Realtime habilitado: `ALTER PUBLICATION supabase_realtime ADD TABLE notifications`
- [ ] Índice de performance: `idx_notifications_user_lida` em `(user_id, lida, created_at DESC)`

### AC 2: Tabela `notification_preferences` criada para customização
- [ ] Tabela `notification_preferences` criada
  - Campos: `id`, `user_id`, `organization_id`, `canal` (email/whatsapp/in-app), `tipo_notificacao`, `habilitado`, `created_at`
- [ ] RLS policies: Usuários podem ver apenas suas próprias preferences
- [ ] UI em `src/pages/NotificationSettings.tsx` para gerenciar preferences

### AC 3: Triggers SQL implementados e testados
- [ ] **Trigger 1**: `trg_contrato_status_change` — Dispara quando status do contrato muda
  - Cria registro em `notifications` para todos os membros ativos da organização
  - Mapeia status para rótulo em português (rascunho → Rascunho, etc)
- [ ] **Trigger 2**: `trg_nova_obrigacao` — Dispara ao inserir nova obrigação contratual
  - Cria notificação incluindo título do contrato e data de vencimento
- [ ] **Trigger 3**: `trg_novo_contrato` — Dispara quando novo contrato é criado
  - Notifica todos os membros sobre novo contrato

### AC 4: Função cron diária para alertas de vencimento
- [ ] Função `job_notificar_vencimentos()` criada e testável
  - Busca contratos vencendo em exatamente 30, 15, 7 dias
  - Evita duplicatas (verifica se já notificou hoje)
  - Cria notificações para todos os membros ativos
- [ ] Agendamento via `pg_cron` (executar às 8h UTC diariamente)
  - Comando: `SELECT cron.schedule('notificar-vencimentos-diario', '0 8 * * *', 'SELECT public.job_notificar_vencimentos()')`

### AC 5: Edge Functions para delivery multi-canal
- [ ] **enviar-notificacao-email** — Resend API
  - Recebe `notification_id` como parâmetro
  - Busca preferences do usuário
  - Envia email se canal=email e habilitado=true
  - Trata erro gracefully (retry logic)
- [ ] **enviar-notificacao-whatsapp** — WhatsApp API
  - Similar ao email, mas para WhatsApp Business API
  - Valida número de telefone
- [ ] **enviar-notificacao-financeiro** — Custom financeiro
  - Notificações específicas para alertas financeiros

### AC 6: UI preferences funcionando end-to-end
- [ ] `src/pages/NotificationSettings.tsx` carrega preferences do BD
- [ ] Usuário consegue:
  - [ ] Selecionar canal preferido (email/WhatsApp/in-app)
  - [ ] Habilitar/desabilitar tipos específicos de notificação
  - [ ] Salvar preferences (PUT para tabela `notification_preferences`)
- [ ] Backend respeita preferences ao enviar notificações

### AC 7: Realtime updates funcionando em UI
- [ ] `src/components/notifications/NotificationPanel.tsx` subscrita em Realtime
  - Quando nova `notification` é inserida para o user, aparece em tempo real
  - Badge de unread count atualiza automaticamente
- [ ] Usuário consegue marcar notificação como lida (UPDATE `lida=true`)

### AC 8: Testes básicos passando
- [ ] Teste: Inserir novo contrato → notificação criada com trigger
- [ ] Teste: Chamar `job_notificar_vencimentos()` → notificações criadas para datas certas
- [ ] Teste: RLS policies respeitam organization_id (usuário de org A não vê notificações de org B)
- [ ] Teste: Realtime subscription recebe atualizações

## Complexity Estimate

**Estimation**: 21 pontos (LARGE)

- Triggers SQL: 5 pontos (3 triggers + função cron)
- Edge Functions: 5 pontos (3 functions, integração APIs externas)

**Risk**: MEDIUM

## Risks & Mitigations

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|--------|-----------|
| Triggers causam lentidão em contratos table | MÉDIA | ALTO | Índices otimizados, testes de performance |
| RLS policies genéricas (não granulares) | BAIXA | MÉDIO | Audit de policies + teste de penetração |
| Edge Functions falham silenciosamente | MÉDIA | MÉDIO | Logging estruturado + alertas de erro |
| Realtime gera muitas conexões | BAIXA | MÉDIO | Connection pooling + rate limiting (future) |
| Duplicate notifications se cron falha | BAIXA | BAIXO | Check `created_at::date` para evitar dups |
