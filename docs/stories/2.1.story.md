## Descrição

Implementar cobertura de testes para `src/pages/Auth.tsx` (19.4KB, ~800 linhas), cobrindo fluxos críticos de autenticação:
- Login via OAuth2 (Google, GitHub, etc)
- 2FA/TOTP validation (SHA-256)
- Session management
- Password validation
- Logout
- Error handling (invalid credentials, MFA failure, etc)

### Contexto

Auth.tsx é um dos componentes mais críticos do sistema — qualquer falha aqui pode:
- ❌ Bloquear usuários de acessar sistema
- ❌ Causar data leaks (session tokens)
- ❌ Permitir acesso não-autorizado

**Impacto**: Sem testes, refactor futuro ou mudanças é impossível. Regressões silenciosas podem passar para produção.

**Business Impact**: Legal liability se falha de auth causar data breach.

---

## Acceptance Criteria

### AC 1: Unit Tests para Login Flow
- [ ] Teste: OAuth2 login redirect funciona
  - Given: Usuário clica em "Login com Google"
  - When: OAuth provider retorna token
  - Then: Session é criada e usuário redirecionado para dashboard
- [ ] Teste: Login com credenciais inválidas falha gracefully
  - Given: Email/password incorretos
  - When: Form é submetido
  - Then: Erro é exibido (no unauthorized)
- [ ] Teste: Password validation funciona
  - Given: Senha < 8 caracteres
  - When: Form é submetido
  - Then: Erro de validação é exibido

**Coverage Target**: 90% de `src/pages/Auth.tsx` (login section)

### AC 2: Unit Tests para 2FA/TOTP
- [ ] Teste: TOTP token gerado corretamente
  - Given: User ativa 2FA
  - When: QR code é scaneado + TOTP token é gerado
  - Then: Token é válido por 30 segundos
- [ ] Teste: TOTP validation funciona
  - Given: Token válido gerado
  - When: Código é inserido em 2FA screen
  - Then: Auth completa com sucesso
- [ ] Teste: Código expirado é rejeitado
  - Given: Token gerou há > 30 segundos
  - When: Código é inserido
  - Then: "Código expirado" erro é exibido
- [ ] Teste: Código inválido é rejeitado
  - Given: Token incorreto é inserido
  - When: Form é submetido
  - Then: Erro é exibido

**Coverage Target**: 85% de TOTP logic em Auth.tsx

### AC 3: Integration Tests para Session Management
- [ ] Teste: Session é criada após login
  - Given: User faz login com sucesso
  - When: Dashboard é acessado
  - Then: `auth.session` existe em Supabase context
- [ ] Teste: Session token é armazenado seguramente
  - Given: User faz login
  - When: Token é armazenado
  - Then: Token NÃO está em localStorage (apenas httpOnly cookie)
- [ ] Teste: Session expira após timeout
  - Given: Session ativa
  - When: Session timeout (24h) é atingido
  - Then: User é deslogado automaticamente
- [ ] Teste: Refresh token funciona
  - Given: Access token expirou
  - When: Refresh token é enviado ao servidor
  - Then: Novo access token é retornado

**Coverage Target**: 75% de session handling

### AC 4: Unit Tests para Logout
- [ ] Teste: Logout limpa session
  - Given: User logado
  - When: Logout é clicado
  - Then: Session é destruída, user redirecionado para login
- [ ] Teste: Tokens são removidos do storage
  - Given: Session com tokens
  - When: Logout é executado
  - Then: Todos os tokens são removidos

**Coverage Target**: 90% de logout flow

### AC 5: Error Handling Tests
- [ ] Teste: Network error é tratado
  - Given: Network indisponível
  - When: Login é tentado
  - Then: "Sem conexão" erro é exibido + retry button
- [ ] Teste: Server error é tratado
  - Given: Supabase retorna 500
  - When: Login é tentado
  - Then: "Erro no servidor" mensagem é exibida
- [ ] Teste: Invalid credentials error
  - Given: Email existe mas password está errado
  - When: Form é submetido
  - Then: "Email ou senha incorretos" (genérico, sem leak info)

**Coverage Target**: 85% de error paths

### AC 6: Mocks Configurados
- [ ] Mock de `@supabase/supabase-js` (client)
  - [ ] `auth.signInWithOAuth()`
  - [ ] `auth.verifyOtp()`
  - [ ] `auth.signOut()`
  - [ ] `auth.refreshSession()`
- [ ] Mock de `use2FA` hook
- [ ] Mock de router (next/router)
- [ ] Mock de localStorage (para token storage)

**Coverage Target**: 100% de dependencies mockadas

### AC 7: Testes Passando em CI/CD
- [ ] Todos os testes passam localmente
  - `npm run test -- src/pages/Auth.tsx`
- [ ] Coverage report gerado
  - `npm run test:coverage -- src/pages/Auth.tsx`
  - Resultado: >= 85% coverage
- [ ] ESLint passing
  - `npm run lint`
- [ ] TypeScript typecheck passing
  - `npm run typecheck`
- [ ] CI/CD pipeline (GitHub Actions) passa
  - Tests executam em paralelo com lint + typecheck

**Coverage Target**: 85% line coverage, 80% branch coverage

### AC 8: Documentation
- [ ] README em `src/__tests__/auth/` documentando:
  - Como rodar testes: `npm run test -- src/__tests__/auth/`
  - Como gerar coverage: `npm run test:coverage`
  - Padrão de testes usado (Arrange-Act-Assert)
  - Fixtures/mocks disponíveis
- [ ] Testes têm comentários explicando fluxos complexos
- [ ] Test names são descritivos (não "test 1", "test 2")
